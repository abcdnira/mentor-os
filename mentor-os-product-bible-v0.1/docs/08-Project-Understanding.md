# 08 - Project Understanding Engine

## 1. 目标

Project Understanding Engine 负责帮助 Mentor 理解用户真实项目，并提炼为面试、简历、能力画像可用的信息。

## 2. MVP 输入方式

MVP 支持：

- 用户手动填写项目说明
- 上传项目 zip
- 粘贴 README
- 粘贴核心代码片段
- 粘贴接口/目录结构

后续支持：

- GitHub/GitLab 仓库
- 自动代码扫描
- AST 分析
- 依赖图
- 业务模块识别

## 3. 不应该直接把整个项目塞给 AI

原因：

- token 成本高
- 上下文有限
- 无关代码太多
- 代码隐私风险
- 总结效果不稳定

正确流程：

```text
项目输入
↓
扫描目录
↓
排除无关文件
↓
识别关键模块
↓
文件级摘要
↓
目录级摘要
↓
模块级摘要
↓
项目经验摘要
↓
简历/面试/能力更新
```

## 4. 输出内容

项目理解后应生成：

- 项目背景
- 技术栈
- 系统架构
- 业务模块
- 用户职责
- 核心难点
- 解决方案
- 技术亮点
- 简历描述
- 面试回答
- 可能追问
- 能力提炼
- MindMap

## 5. 示例：IM 项目

输出：

```markdown
# IM 系统项目画像

## 项目背景
面向社交场景的即时通讯系统，支持单聊、群聊、红包、客服等能力。

## 技术栈
Golang / WebSocket / Redis / MQ / MySQL / MongoDB

## 核心模块
- 长连接管理
- 消息投递
- 离线消息
- 群聊
- ACK
- 红包消息
- 客服消息

## 面试亮点
该项目能体现用户在高并发、消息可靠性、Redis/MQ、复杂业务建模方面的能力。
```

## 6. 项目图谱

项目应形成图谱：

```text
IM
├── WebSocket
├── 消息可靠性
│   ├── ACK
│   ├── 重试
│   └── 幂等
├── 群聊
│   ├── 写扩散
│   └── 读扩散
└── 红包
    ├── 分布式锁
    └── 幂等
```
